《应届生 AI 求职伴侣》聊聊功能 MVP 实现 PRD v3.0

一、文档信息

产品名称：应届生 AI 求职伴侣
当前迭代目标：先跑通“聊聊”功能，验证大模型对话、全局聊天入口、本地记忆与用户画像更新链路
产品定位：面向应届生 / 校招生 / 毕业 1 年内求职者的 AI 求职伴侣
文档版本：v3.0
文档日期：2026-04-28
适用阶段：开发实现阶段 / MVP 阶段

二、文档目的

本 PRD 用于指导当前阶段的开发落地。与前期偏愿景和完整产品规划的文档不同，本版本只聚焦一个目标：

先把“聊聊”功能跑通。

本阶段需要验证以下能力是否成立：
1. 用户能在首页通过开放式问题开始对话
2. 用户能在任意功能页通过常驻入口随时聊天
3. 后端能统一调用大模型
4. 后端能读取并更新本地记忆文件
5. 系统能持续形成用户画像和求职状态
6. 多轮对话不会丢失上下文

三、产品背景

应届生求职过程中，最核心的并不只是简历修改，而是方向、认知、行动和陪伴。用户往往在以下几个问题上持续卡住：
- 不知道自己适合什么岗位
- 不知道自己能做什么
- 不知道下一步先干什么
- 求职过程焦虑、挫败、无助

因此，本产品的最底层能力不应该只是“问答”，而应该是：
- 大模型对话与分析
- 本地长期记忆
- 持续追踪用户求职状态

当前版本只做最小闭环，不追求一次性做完全部功能。

四、MVP 目标

1. 功能目标
- 跑通首页 onboarding 聊天
- 跑通全局常驻聊天入口
- 跑通大模型后端统一调用
- 跑通本地记忆读写
- 跑通用户画像增量更新

2. 体验目标
- 用户在任何页面都能随时发起聊天
- 首页能通过开放式问题建立初印象
- 系统能逐步理解用户，而不是每次重新开始
- 聊天结果要让用户感到被理解、有建议、有下一步

3. 技术目标
- 后端作为唯一模型调用入口
- Prompt 可按场景拆分
- 模型输出支持结构化结果
- 记忆文件可持久化、可读取、可更新

五、范围定义

1. 本期做的内容
- 首页开放式问答 onboarding
- 任意功能页常驻聊天入口
- 大模型后端调用
- 用户画像文件
- 求职状态文件
- 聊天历史文件
- 记忆更新日志
- 用户可查看基础画像

2. 本期不做的内容
- 多模型路由
- 复杂知识库检索
- 高级情绪分类
- 长篇复盘报告
- 推荐算法
- 社交功能
- 团队协作功能
- 复杂权限系统

六、产品原则

1. 聊天优先
聊天不是独立功能页，而是贯穿所有页面的全局能力。

2. 首页建模
首页不是展示页，而是用户画像的起点。

3. 记忆可控
记忆以本地优先、用户可查看和可清除为原则。

4. 输出结构化
模型输出不仅要有自然语言回复，还要有结构化字段供系统处理。

5. 先跑通再完善
本期重点是闭环跑通，不追求一次性做大做全。

七、用户与场景

1. 核心用户
- 应届毕业生
- 研究生毕业求职者
- 毕业 1 年内仍在求职或转方向的用户

2. 首期高频场景
- 首页首次进入，系统通过开放式问题建立第一印象
- 用户在岗位匹配、简历优化、面试模拟等页面中随时提问
- 用户对结果继续追问
- 用户表达焦虑、困惑、吐槽，系统承接并记录

八、核心用户流程

流程 A：首页 onboarding
1. 用户首次进入首页
2. 系统显示欢迎语和开放式问题
3. 用户自由回答
4. 系统总结用户表达并追问一个关键问题
5. 系统形成初始画像
6. 写入本地记忆文件
7. 用户继续对话或进入功能页

流程 B：功能页常驻聊天
1. 用户进入任一功能页
2. 页面右下角显示常驻聊天入口
3. 用户点击打开聊天面板
4. 系统读取当前页面上下文 + 历史记忆
5. 用户发起问题
6. 后端调用大模型返回结果
7. 系统自动判断是否需要更新记忆
8. 若需要，则写入本地记忆文件

流程 C：持续追踪
1. 用户多次使用产品
2. 系统逐渐积累用户画像、状态和偏好
3. 下一次聊天时，系统自动引用历史认知
4. 用户感受到系统“越来越懂我”

九、信息架构

1. 首页
- 欢迎说明
- 开放式对话入口
- 初始画像生成
- 进入功能页的入口

2. 功能页
- 页面主功能区域
- 常驻聊天入口
- 聊天抽屉 / 侧边面板
- 与当前页面联动的上下文提示

3. 我的求职画像页
- 已知信息
- 当前阶段
- 待确认信息
- 编辑 / 清除 / 仅本次生效控制

十、页面与交互需求

1. 首页 onboarding 页面

目标
通过 1~3 轮开放式问答，建立用户对产品的第一印象，并生成基础画像。

页面内容
- 产品标题
- 简短说明
- 1 个开放式主问题
- 用户输入框
- 系统总结区
- 下一步追问区

推荐问题示例
- 你现在最想解决的求职问题是什么？
- 你目前更关注哪些岗位方向？
- 你最担心的事情是什么？
- 你希望我更像分析助手，还是更像陪你一起行动的伙伴？

交互要求
- 一次只问一个核心问题
- 回答后先总结，再决定是否继续追问
- 允许用户跳过
- 不要像问卷

2. 全局聊天入口

目标
让用户在任意功能页都可以快速对话。

交互要求
- 所有功能页显示同一个常驻入口
- 推荐形态为右下角悬浮按钮
- 点击后展开聊天抽屉或侧边面板
- 不打断当前页面主流程
- 面板中展示当前页面上下文标签

3. 我的求职画像页

目标
让用户查看系统已经理解了什么，并允许修正。

展示内容
- 基础信息
- 目标岗位
- 能力优势
- 当前卡点
- 当前情绪状态
- 待确认信息

操作能力
- 编辑
- 清除
- 仅本次生效

十一、聊聊功能 MVP 定义

1. MVP 核心能力
- 接收用户输入
- 识别当前场景
- 读取用户画像与状态
- 组合 prompt 调用大模型
- 返回自然语言回复
- 识别是否有记忆更新
- 更新本地文件

2. MVP 成功标准
- 首页可以连续对话
- 任意页面可呼出聊天入口
- 聊天过程中上下文不丢失
- 模型响应稳定
- 记忆文件可成功读写
- 用户画像可持续更新

十二、记忆系统设计

1. 记忆文件结构建议

目录结构：
- data/users/{user_id}/profile.json
- data/users/{user_id}/state.json
- data/users/{user_id}/chat_history.json
- data/users/{user_id}/memory_log.json

2. 文件职责

profile.json
- 存用户长期稳定画像
- 如学校、专业、目标岗位、偏好、能力、约束

state.json
- 存当前求职状态
- 如当前阶段、当前问题、最近行动、当前情绪

chat_history.json
- 存原始对话记录
- 便于回看、调试和补上下文

memory_log.json
- 存记忆变更日志
- 便于追踪历史修改和纠错

3. 记忆更新原则
- 用户明确表达优先于模型推断
- 稳定偏好可写入长期记忆
- 临时情绪不要写成长期事实
- 纠错信息优先覆盖旧认知
- 每次更新要记录来源和置信度

十三、大模型调用设计

1. 调用原则
- 统一由后端调用大模型
- 前端不直连模型接口
- 后端负责拼接上下文、处理结果和更新记忆

2. 后端需要输入的内容
- 当前用户消息
- 当前页面信息
- 当前场景类型
- 用户画像摘要
- 求职状态摘要
- 历史对话摘要
- 系统总 Prompt
- 场景 Prompt

3. 模型返回格式要求
建议返回结构化 JSON，包含：
- reply：自然语言回复
- summary：本轮总结
- follow_up_question：下一步追问
- memory_action：是否更新记忆及更新内容
- user_state：用户情绪与阶段判断

十四、Prompt 设计要求

1. 总系统 Prompt
定义角色、目标、边界与回答原则。

2. 首页 onboarding Prompt
用于开放式提问和初始画像生成。

3. 功能页聊天 Prompt
用于结合当前页面上下文回答。

4. 记忆抽取 Prompt
用于从本轮对话中提炼可写入长期记忆的信息。

十五、后端实现方案

1. 请求链路
前端消息 -> 聊天接口 -> 读取记忆 -> 组装 Prompt -> 调用模型 -> 解析结果 -> 写入记忆 -> 返回前端

2. 推荐接口

POST /api/chat
- 用户发起聊天的统一入口

GET /api/profile/:user_id
- 获取用户画像和状态

POST /api/profile/update
- 用户手动修改画像或记忆

3. 服务模块划分
- controller：接收请求和返回响应
- service：处理聊天业务逻辑
- prompt builder：拼装 prompt
- llm client：调用大模型
- memory manager：读写本地文件
- parser：解析模型结构化输出

十六、接口请求与响应建议

1. POST /api/chat 请求示例
{
  "user_id": "u_001",
  "session_id": "s_001",
  "current_page": "home",
  "scene": "onboarding",
  "message": "我现在对找工作很迷茫，不知道自己适合什么。"
}

2. POST /api/chat 响应示例
{
  "reply": "没关系，我们先一步步来。我先了解一下你最近接触过哪些方向？",
  "summary": "用户目前对求职方向感到迷茫。",
  "follow_up_question": "你最近接触过哪些岗位方向？",
  "memory_updated": true,
  "user_state": {
    "emotion": "anxious",
    "stage": "direction_exploration"
  }
}

十七、非功能性要求

1. 安全性
- API Key 不暴露给前端
- 用户记忆文件仅保存求职相关内容
- 输入和输出需要基础校验

2. 可维护性
- Prompt 与业务逻辑分离
- 场景与通用能力分离
- 记忆文件与聊天记录分离

3. 稳定性
- 模型超时需要兜底
- 调用失败时要返回友好提示
- 不允许一个请求影响整个记忆文件写入

4. 可扩展性
- 未来可扩展多模型
- 未来可扩展更完整画像页
- 未来可扩展更细的分析能力

十八、MVP 开发优先级

P0：必须完成
- 首页 onboarding 聊天
- POST /api/chat 接口
- 本地用户记忆文件
- 结构化模型输出
- 聊天记录持久化

P1：尽快完成
- 任意功能页常驻聊天入口
- 画像查看页
- 记忆更新日志
- 用户手动编辑画像

P2：后续优化
- 更丰富的上下文摘要
- 更细的状态判断
- 更强的情绪陪伴能力

十九、测试验收标准

1. 功能验收
- 首页可以完成至少 3 轮自然对话
- 功能页可随时打开聊天面板
- 聊天记录可正确保存
- 画像可正确读取和更新
- 用户纠正信息后能覆盖旧记忆

2. 体验验收
- 用户不会明显感觉“每次都从零开始”
- 用户能感受到系统在持续理解自己
- 用户能获得明确、可执行的下一步建议

3. 技术验收
- 模型调用成功率稳定
- 记忆文件写入成功率稳定
- 结构化输出可被后端正确解析

二十、风险与应对

1. 风险：Prompt 太长导致输出不稳定
应对：拆分系统 Prompt、场景 Prompt 和记忆抽取 Prompt。

2. 风险：记忆写入过多，污染画像
应对：只写稳定信息，临时情绪只写状态不写画像。

3. 风险：上下文过长导致成本高
应对：保留最近对话 + 历史摘要 + 用户画像摘要。

4. 风险：用户不信任记忆机制
应对：提供画像查看、编辑、清除能力。

二十一、结论

本版本 PRD 的核心目标不是做完整产品，而是先把聊天中枢跑通。

只要以下 4 件事成立：
- 能聊
- 能记
- 能追踪
- 能在任意页面随时聊

这个产品就具备了从“工具”走向“求职伴侣”的底层基础。

二十二、下一步开发建议

建议开发顺序如下：
1. 搭建 /api/chat 接口
2. 实现本地 user profile / state / history 文件结构
3. 接入单一大模型调用
4. 完成首页 onboarding 问答
5. 完成功能页悬浮聊天入口
6. 实现记忆抽取与更新
7. 增加画像查看页

本阶段的成功标志不是功能多，而是闭环稳。
